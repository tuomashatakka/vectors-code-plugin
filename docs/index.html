vectors-plugin

Local semantic search over your code & docs, indexed by project.

A global, on-disk RAG store partitioned by project. Point it at a repo or a folder of notes; get reranked semantic search back — per project or across all of them. Ships as a Claude Code / opencode / Claude Desktop plugin with an MCP server. No API keys, no network at query time.

What it is

One store on disk holds many projects; each is its own embedded, cross-encoder-reranked index. The active project is auto-resolved from your working directory, so an agent working inside a repo gets that repo's retrieval automatically — and you can still search globally across every project at once. 

$VINDEX_HOME            one global RAG database
  ├── scene/            a project — its own collection + config + root
  ├── portfolio/
  └── rustbook/

per-project   files → chunk → embed → store        (ingest, once)
              query → embed → top-k → rerank        (search, fast, offline)
global        fan out across projects → merge → one rerank → tagged hits

Core ideas

Project from your cwd

Resolution walks $VINDEX_PROJECT → nearest project root → a .git/.vindex marker → default. You rarely name the project.

Per-project & global search

Search one project, or fan out across all of them and rerank the union so results are comparable even across different embedding models.

Fully local

Embeddings (all-MiniLM-L6-v2) and the cross-encoder reranker run on-device. No API keys; no network when you query.

Any corpus

Local dirs or git repos, filtered by globs, with markdown / code / text chunking chosen per file. Public URLs reconstructed for each hit.

Four ways to use it

CLI
Create projects, ingest, run one-off queries (project or global), list projects, serve the viewer — vindex.py.
MCP server
Live tools for Claude / agents: search, search_global, current_project, ingest, create_project, and more.
Library
import vector_index — the Project class plus global_search and resolve_project_name.
3D viewer
A three.js “synapse” navigator that PCAs a project's embedding space and lets you explore it interactively.

Every project ships a 3D “synapse” viewer that PCAs the embeddings to three dimensions and links nearest neighbours. Below is a live, procedural demo — drag to orbit, scroll to zoom, type to search. Quickstart

cd skills/vector-index && bash setup.sh        # build the venv, install deps
PY=./.venv/bin/python

# index the project you're standing in (root defaults to --source)
$PY scripts/vindex.py create scene --source . --strategy code \
    --glob '**/*.ts' --glob '**/*.md'
$PY scripts/vindex.py ingest
$PY scripts/vindex.py query "how does the flock pick ideas?"

# ask across every project at once
$PY scripts/vindex.py search "welded indexed geometry deterministic seed"

Unified knowledge database design

The repo includes a full design + DDL for consolidating everything into a single local PostgreSQL + pgvector store, so vectors and memory live side by side and cross-reference cleanly.

  • One store, four kinds of knowledge — chunk vectors, chat history, external references (URLs, Drive, Notion), and your own document/codebase content.
  • Multi-project, not isolated — a project hierarchy plus sibling links; content is tagged by project yet still searchable globally.
  • A 4-level memory ladder — from verbatim content (L0) up to vague concepts (L3), derived by a local model and regenerated when the source changes — without ever mutating the originals.
  • Constantly-learning memory — durable facts and preferences with confidence, reinforcement, and decay.
  • Token-saving retrieval — serve high-level summaries first and drill to detail only on demand.

Background daemon & backup in repo

A single long-lived process (launchd on macOS, systemd on Linux) keeps the database current and recoverable, with no manual steps.

  • Chat feeder — watches transcript files and ingests new conversation context automatically.
  • Source feeder — mirrors changed project files into the store, content-hash diffed.
  • Digest worker — drains a job queue, embedding content and running local-Ollama summarization, fact, and reference extraction.
  • Optional daily backup — pg_dump pushed to OneDrive or Google Drive (local sync folder or rclone), an Obsidian vault, and/or a Notion manifest, with retention.

Everything runs locally — no API keys, no query-time network.
